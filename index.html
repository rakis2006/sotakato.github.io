スライム大戦争 (Processing)

制作時期：2024年 冬（大学1年次 最終課題）

使用技術：Processing（Javaベースの視覚表現用環境）

作品詳細

「にゃんこ大戦争」のゲームシステムを参考に制作した、横スクロール型のタワーディフェンスゲームです。 自陣からスライムを召喚し、敵陣の耐久値をゼロにすることで勝利となります。 キャラクターのステータス管理、攻撃判定、資金のリソース管理など、ゲームの根幹となるシステムをで実装しました。

工夫した点

  • オブジェクト指向による効率的な管理：
    味方・敵の全てのキャラクターを「Characterクラス」として抽象化し、継承を利用することで、異なるステータスや挙動を持つ多彩なスライムを最小限のコードで実装しました。
  • ゲームバランスの追求：
    召喚コスト、攻撃力、移動速度の数値を細かく調整し、単調な連打ゲームにならないよう、戦略性を持たせるチューニングを行いました。
  • 描画負荷の最適化：
    大量のキャラクターが画面上に現れても処理が重くならないよう、配列やArrayListを活用して不要になったオブジェクトを適切にメモリから解放する処理を工夫しました。

書籍管理システム (Webアプリ)

制作時期：2025年 3月（3日間 短期インターン）

使用技術：Java, MySQL

プロジェクト概要

「おすすめの書籍を共有する」Webアプリの機能拡張とUI改善をチームで行いました。 大阪のメンバーとリモートで連携し、実務と同様の「要件定義」「設計概要の作成」から実装までを担当しました。

主な実装・改善内容

  • 検索・並び替え機能の高度化：
    書籍名による絞り込み検索に加え、発売日の「新しい順・古い順」の切り替え機能を実装し、データ抽出の利便性を向上させました。
  • 入力体験(UX)の向上：
    書籍追加時の「タグ機能」や、エラー時の即時フィードバック（バリデーション）、削除前の確認モーダル表示など、ユーザーが迷わないUI/UXを追求しました。

データベース（MySQL）を用いた実装

データの永続化のためにMySQLを使用し、効率的なデータ操作を意識して実装しました。

  • 高度なデータ抽出：
    書籍名による絞り込み検索や、発売日の「新しい順・古い順」のソート機能を実装。SQLのクエリを意識し、大量のデータから必要な情報を正確に取得できるよう設計しました。

チーム開発とコミュニケーションの重要性

3日間という限られた期間、かつリモート環境での開発を通じ、「素早く正確なコミュニケーション」がプロジェクトの成否を分けることを痛感しました。

  • 認識のズレを防ぐ要件定義：
    実装に入る前に、タイトル、著者名、タグといった入力項目の文字制限やバリデーション（入力チェック）条件を細かく言語化し、チーム全員で共通認識を持ちました。
  • リモート環境でのスピード感：
    対面ではないからこそ、進捗状況の共有や技術的な相談をzoomで即座に行い、詰まっている時間を最小限にするよう努めました。この素早い連携により、3日間という短期間で高品質なシステムを完遂できました。

飲食系モバイルオーダー UIデザイン

制作時期：2025年 7月（大学2年次 デザイン講義最終課題）

使用ツール：Figma

プロジェクト詳細

実在する飲食店「日高屋」のモバイルオーダー画面をモデルに、既存の課題を分析し、「より直感的で注文ミスを防ぐデザイン」をコンセプトにプロトタイプを作成しました。 単なる模倣ではなく、実際の利用シーン（店舗の照明、客層、操作手順）を想定した再設計を行いました。

工夫した点（UXの意識）

  • 視認性の徹底：
    店内の照明やスマホの明るさに左右されず、年配の方でも見やすいよう、コントラスト比の調整やフォントサイズの最適化を行いました。
  • 注文フローの簡略化：
    「トッピング選択からカート投入まで」のタップ回数を最小限に抑え、現在の注文状況が常に一目でわかるプログレス表示を取り入れました。

苦労した点

「情報量」と「スッキリ感」の両立：
メニュー数が多い中で、写真の魅力とお得なセット情報をどう配置するかに苦労しました。Figmaのコンポーネント機能を活用して複数のパターンを作成し、実際に自分のスマホで操作しながら、指が届きやすい範囲（親指の可動域）に基づいたボタン配置を追求しました。

Todo管理モバイルアプリ (Flutter,Firebase)

制作時期：2025年 10月（約1ヶ月間）

使用技術：Dart (Flutter), Firebase (Cloud Firestore)

プロジェクト詳細

1ヶ月間のFlutter勉強会にて制作。教材はアラビア語の動画でしたが、単なる書き写しで終わらせず、「動画を見ずにゼロから自力で実装できるまで」、プロジェクトを何度も作り直して構造を体に叩き込みました。

習得へのプロセスと工夫

  • 徹底した反復練習：
    最初は書き方を模倣するだけでしたが、2回目、3回目とゼロから作り直すことで、Flutter特有のWidget構成やデータの流れを深く理解しました。最終的には外部の解説に頼らず、ロジックを自力で構築できるまで精度を高めました。
  • FirebaseによるリアルタイムDB実装：
    未経験からFirebaseとの連携に挑戦。何度もエラーに直面しながらも、メンターへの質問と自己学習を繰り返し、データのCRUD操作（追加・読み取り・更新・削除）を完璧に習得しました。

この経験から得たもの

未知の技術であっても、「理解できるまで反復し、自分のものにする」という学習の型を確立できました。言語の壁や未経験という壁を、圧倒的な作業量で乗り越えた経験は、現在の開発スタイルの礎となっています。

AI恋人アプリ「夢カノ」（ハッカソン初挑戦作品）

制作時期：2025年 12月（10日間）

担当役割：チームリーダー / フロントエンド開発

プロジェクト詳細とリーダーとしての取り組み

3人チームで挑んだ初のハッカソンです。当初はリモート中心の連携不足により進捗が滞っていましたが、私がリーダーとなりタスクの絞り込みとタスクの再割り当てを徹底したことで、開発スピードを大幅に向上させました。

直面した課題と苦渋の決断

フロントエンドを担当しつつ全体の進行を管理していましたが、対面作業に切り替わった際に、データベース側のタスクができたと思っていたができていなかったということがあり、予定通りの完成が危ぶまれる状況に陥りました。私は「未完成で終わらせない」ために機能の削減を提案し、完成を最優先する行動を促しましたが、一番大事な画像生成をユーザーができるようにするという部分ができませんでした。

ハッカソンを通じて得た教訓と改善点

  • コミュニケーションの密度：
    「できているだろう」という思い込みを排除し、リモート環境でも細かく進捗を可視化することの重要性を痛感しました。
  • トラブルを想定したバッファ設計：
    機器故障などの不測の事態を含め、工数見積もりには余裕を持たせるべきだと学びました。
  • 技術的な反省：
    バックエンドとフロントエンドの連携部分で、早い段階でプロトタイプを作りそれを拡張していくような開発を進めておけば、より確実で、スムーズに統合できたという改善点を見つけました。

結果として完全な形でのデプロイには至りませんでしたが、この「本気の失敗」から得たプロジェクト管理は、今の開発に大きく活きています。

プログラミング言語 難易度分析システム

制作時期：2026年 1月（大学2年次 データエンジニアリング最終課題）

使用技術：Java, Google Custom Search API, Gemini API

プロジェクト概要

新しいプログラミング言語を学ぶ学習者のために、ネット上の開発者のリアルな声を収集・分析し、学習難易度を可視化するシステムです。 Google検索APIで収集したネット上の開発者の生の情報を、Gemini APIに渡して多角的に分析させる「APIマッシュアップ」を実装しました。

技術的な工夫点（データエンジニアリング）

  • APIマッシュアップの実装：
    単一のAPIでは完結しない「情報の取得（Google）」と「意味の抽出（AI）」という2つの工程をJavaプログラム上で統合し、付加価値の高い情報を生成する仕組みを構築しました。
  • AIの出力結果の定義の厳格化：
    AIの回答に「余計な挨拶」や「表記揺れ」が含まれないよう、responseSchemaを用いてJSON形式の構造を定義。スコアを整数（Integer）として受け取れるようにし、プログラム側での確実なデータ処理を実現しました。
  • 実用的なアウトプットの設計：
    「10段階の難易度スコア」に加え、初心者にも配慮した「子供でも分かる解説」を生成させるなど、ユーザーのターゲット層に合わせたシステムにしました。

この課題で得たもの

API連携による価値創造の基礎：
既存のサービス（検索エンジンとAI）を組み合わせ、元のデータにはなかった「分析」という付加価値を与えるプロセスの面白さを学びました。これにより、ゼロから全てを作るだけでなく、既存の技術を統合して課題を解決する「マッシュアップ」の有用性を実感しました。

AIの解答を欲しいものにできるように意識した設計：
「AIの返答」という不確定な要素を、プログラムが扱える「厳格なデータ」へ変換する重要性を学びました。エラーを未然に防ぐためのデータ型定義やURLエンコードの実装を通じ、システム設計の基礎を習得できました。

データ活用からキャリア支援への視点：
収集したデータをどう見せるかというUXの観点に加え、今後は「市場価値（年収）」などの外部データとさらに連携させ、技術を社会的なニーズと結びつけて考える「サービス開発の視点」を得ることができました。

グラビティ・ランナー

制作時期：2026年 1月（大学2年次 感性情報イメージング最終課題）

使用技術：TypeScript,HTML,CSS

プロジェクト概要

重力を反転させて障害物を避けるランアクションゲームです。「20秒間耐えて女王を救い出す」という明確な目標に対し、難易度が時間が経つごとに上昇する仕組みを実装しました。 初めてのTypeScriptを用いた開発で、実際にWebサイトとして公開可能な形まで構築しました。

技術的な工夫点（設計と実装）

  • オブジェクト指向による継承設計：
    抽象クラスGameObjectを定義し、プレイヤーや障害物へ機能を継承させる設計を行いました。これにより、当たり判定（AABB法）や更新処理の共通化を実現し、拡張性の高いコード構造にしました。
  • 独立した物理エンジン（PhysicsWorld）：
    重力の状態を管理する専用クラスを作成。単なる「上下移動」ではなく、加速度を用いた物理演算を行うことで、重力反転時の「浮遊感」や「着地」の滑らかさを追求しました。
  • 感性を刺激するゲームデザイン：
    経過時間に合わせて背景色が変化し、障害物のスピードが上昇する「緊張感の演出」を実装。スペースキーによる空中静止（線形補間を用いた座標計算）など、操作の楽しさを数学的に計算して実装しました。
  • 永続的なユーザー体験：
    Web Storage APIを利用し、ブラウザを閉じてもハイスコアが保持される機能を搭載。単発のプログラムではなく、一つの「プロダクト」としての完成度を目指しました。

この課題で得たもの

未知の技術に対する「自走力」の獲得：
初めてのTypeScriptに挑戦する際、単にコードを書き写すのではなく、「なぜここでエラーが出るのか」「JavaScriptと何が違うのか」という背後にある法則性を探ることを大切にしました。最初は厳格な型定義に苦戦しましたが、公式ドキュメントを読み解きながらその意図を理解することで、中盤からは言語の特性を味方につけたスムーズな開発ができるようになりました。この経験を通じて、未知の言語であっても自ら学び、使いこなしていく「自走力」に大きな自信がつきました。

怠けてしまう人のための「Todo日記アプリ」

制作時期：2026年 1月（大学2年次 GUIプログラミング最終課題）

使用技術：Java

プロジェクト概要

日々のタスク管理と日記を一体化させ、活動を振り返りながら自己成長を促すデスクトップアプリです。 単なるリスト表示にとどまらず、タスクの達成率をリアルタイムでグラフ化し、過去7回分の推移を可視化することで、モチベーションを維持する工夫を凝らしました。

技術的な工夫点

  • 独自描画によるデータの視覚化：
    外部ライブラリに頼らず、JavaのpaintComponentを拡張して円グラフと棒グラフを自作実装しました。数値計算に基づいて角度や座標を制御し、視覚的に分かりやすいUIを実現しました。
  • イベント駆動型プログラミング：
    ボタンクリックやタブの切り替え、ウィンドウ終了時の確認ダイアログなど、多様なイベントリスナーを適切に配置。ユーザーの操作に対して、システムが意図通りに反応する設計を徹底しました。
  • ファイル入出力による履歴管理：
    PrintWriterやBufferedReaderを活用し、日記データと達成率をテキストファイルに蓄積・復元する機能を実装。アプリを終了してもデータが失われない「実用性」を追求しました。

この課題で得たもの

基礎技術の組み合わせによる問題解決：
Javaの基礎文法を学ぶ段階から、それを組み合わせて「実際に役立つ道具」へと昇華させる難しさと楽しさを学びました。特に、画面（UI）とデータ処理（ロジック）を切り離して考えることの重要性を、開発を通して実感できました。

ユーザー体験を意識した機能追加：
タスクがすべて完了した際に「パーフェクト！」というメッセージを表示するロジックを組み込むなど、使う人の気持ちを考えた小さな工夫が、システムの価値を高めることを学びました。

学内コミュニティ形成支援マッチングアプリ（チーム開発）

プロジェクト期間：2025年 11月 〜 現在進行中

役割・担当：フロントエンド開発リーダー

使用技術：Flutter (Dart), HTML, CSS, Git

[ 2026年 春 学内リリース予定 ]
※現在開発中のため、スクリーンショットの公開を控えております。

プロジェクト概要

大学内において、同じ趣味や目標を持つ学生同士が繋がれるマッチングプラットフォームです。 勉強会で出会った3名のメンバーでチームを結成し、「孤立する学生をゼロにする」という目標のもと、企画・設計・実装を進めています。私はフロントエンド全般の責任者として、ユーザーが直感的に操作できるインターフェースの構築を担当しています。

技術的な工夫点（チーム開発としての視点）

  • マルチプラットフォームを意識したFlutter開発：
    学生がスマホとPCの両方からシームレスにアクセスできるよう、Flutterを採用。各デバイスの画面サイズに最適化された「レスポンシブデザイン」をHTML/CSSの知識も併用しながら実装しています。
  • API仕様の先行定義と調整：
    バックエンド担当者と密に連携し、Swagger等を用いてAPI仕様を事前に固めることで、開発のフロント・バック分離を推進。効率的な並行開発をリードしています。
  • リリース品質のバリデーション実装：
    不特定多数の学生が利用することを想定し、フォーム入力のバリデーションや例外処理を徹底。「ユーザーが誤った操作をしても壊れない」堅牢なフロントエンドを目指しています。

このプロジェクトを通じて学んでいること

チーム開発における「伝える力」の重要性：
個人開発とは異なり、技術的な意図を言葉で正しく伝え、メンバーと認識を合わせることが開発速度に直結することを強く実感しています。GitHubを用いたコード管理や会議を通じ、チームで一つの目標に向かうプロセスを学んでいます。

プロダクトを世に出す責任感：
「自分が動かして満足する」段階を超え、実際のユーザー（同じ大学の学生）の手に渡ることを前提とした品質基準での開発を行っています。リリースに向けたデバッグや品質管理を通じ、エンジニアとしての責任感を養っています。
